Getting Started

Create your first comprehensive Agent Skill in 10 minutes

What You'll Build

In this tutorial, you'll create a comprehensive skill called mvvm-architecture-enforcer that teaches Copilot how to enforce MVVM architecture rules in WinUI 3 applications. This example demonstrates all Agent Skills features including scripts, references, and templates.

Time Required: 10 minutes
Prerequisites: A text editor and access to your project folder
Skill Features: ‚úÖ SKILL.md + ‚úÖ scripts/ + ‚úÖ references/ + ‚úÖ templates/

Why this skill? MVVM architecture violations are the #1 issue in WinUI 3 projects. This skill prevents common mistakes like ViewModels calling DAOs directly, missing partial classes, or using runtime {Binding} instead of {x:Bind}.

Step 4: Add Main Instructions

After the frontmatter, add the core instructions Copilot will follow:

# MVVM Architecture Enforcer

This skill enforces MVVM architecture rules for WinUI 3 / .NET 8 desktop applications.

## When to Use This Skill

Activate this skill when:
- Creating new ViewModels, Services, or DAOs
- Modifying existing MVVM components
- Reviewing code for architecture violations
- Creating XAML views with data binding
- Implementing business logic layers

## CRITICAL ARCHITECTURE RULES

### ‚ùå FORBIDDEN - Never Do These

1. **ViewModels calling DAOs directly** - MUST go through Service layer
2. **Static DAO classes** - All DAOs MUST be instance-based
3. **Runtime {Binding} in XAML** - Use compile-time {x:Bind} only
4. **Business logic in .xaml.cs code-behind** - Belongs in ViewModel or Service
5. **DAOs throwing exceptions** - Return Model_Dao_Result with error details

### ‚úÖ REQUIRED - Always Do These

1. **MVVM Layer Flow:** View (XAML) ‚Üí ViewModel ‚Üí Service ‚Üí DAO ‚Üí Database
2. **ViewModels:** Partial classes inheriting from ViewModel_Shared_Base
3. **Services:** Interface-based with dependency injection
4. **DAOs:** Instance-based, return Model_Dao_Result
5. **XAML Bindings:** Use x:Bind with explicit Mode (OneWay/TwoWay/OneTime)

## Quick Validation

Before accepting any code, check:

- [ ] ViewModel is a partial class
- [ ] ViewModel inherits from ViewModel_Shared_Base or ObservableObject
- [ ] ViewModel does NOT call DAOs directly (must use Services)
- [ ] Services have interfaces (IService_*)
- [ ] DAOs are instance-based (not static)
- [ ] XAML uses x:Bind (not Binding)
- [ ] Async methods end with "Async" suffix

## Need More Details?

For comprehensive rules: See [architecture-rules.md](../../.github/skills/mvvm-architecture-enforcer/references/architecture-rules.md)  
For common mistakes: See [common-violations.md](../../.github/skills/mvvm-architecture-enforcer/references/common-violations.md)  
For validation script: Run [validate-mvvm.ps1](../../.github/skills/mvvm-architecture-enforcer/scripts/validate-mvvm.ps1)  
For code templates: Use [viewmodel-template.cs](../../.github/skills/mvvm-architecture-enforcer/templates/viewmodel-template.cs)

Step 5: Create the Validation Script

Create scripts/validate-mvvm.ps1 to check code for violations:

# validate-mvvm.ps1
[CmdletBinding()]
param(
    [Parameter()]
    [string]$Path = ".",

    [Parameter()]
    [switch]$PassThru
)

Write-Verbose "Validating MVVM architecture under path: $Path"

$violations = New-Object System.Collections.Generic.List[string]

$viewModelFiles = Get-ChildItem -Path $Path -Recurse -File -Filter "*.cs" |
    Where-Object { $_.Name -match "ViewModel" }

foreach ($file in $viewModelFiles) {
    $content = Get-Content -Path $file.FullName -Raw

    if ($content -match "Helper_Database_") {
        $violations.Add("$($file.FullName): ViewModel references Helper_Database_* (forbidden)")
    }

    if ($content -match "\bDao_\w+\b") {
        $violations.Add("$($file.FullName): ViewModel references Dao_* directly (likely violation)")
    }

    if ($content -notmatch "\bpartial\s+class\b") {
        $violations.Add("$($file.FullName): ViewModel is missing 'partial class'")
    }
}

$xamlFiles = Get-ChildItem -Path $Path -Recurse -File -Filter "*.xaml"

foreach ($file in $xamlFiles) {
    $content = Get-Content -Path $file.FullName -Raw

    if ($content -match "\{Binding\s") {
        $violations.Add("$($file.FullName): XAML uses {Binding} (forbidden; use {x:Bind})")
    }
}

if ($violations.Count -eq 0) {
    Write-Output "No MVVM violations found."
    return
}

Write-Warning "Found $($violations.Count) potential MVVM violation(s)."

foreach ($v in $violations) {
    Write-Output $v
}

if ($PassThru.IsPresent) {
    Write-Output ([PSCustomObject]@{
        ViolationCount = $violations.Count
        Violations     = $violations
    })
}

Why include a script? Scripts provide deterministic validation that complements Copilot's analysis. They're also useful for CI/CD pipelines.

Step 6: Add Reference Documentation

Create references/architecture-rules.md for detailed guidance:

# MVVM Architecture Rules (Detailed)

## Layer Separation

### View Layer (XAML)
- Contains UI markup only
- Uses x:Bind for compile-time binding
- No business logic in code-behind (.xaml.cs)

### ViewModel Layer
- Partial classes inheriting from ViewModel_Shared_Base
- Uses [ObservableProperty] and [RelayCommand] attributes
- Calls Services (never DAOs directly)
- Handles UI state (IsBusy, StatusMessage)

### Service Layer
- Interface-based (IService_*)
- Orchestrates DAOs and business logic
- Registered as Singleton in DI container
- Handles error responses from DAOs

### DAO Layer
- Instance-based (never static)
- Returns Model_Dao_Result
- Never throws exceptions
- Uses stored procedures for MySQL

## Data Flow

```
User Interaction
    ‚Üì
View (XAML)
    ‚Üì (x:Bind to Command)
ViewModel (RelayCommand method)
    ‚Üì (calls)
Service (business logic)
    ‚Üì (calls)
DAO (data access)
    ‚Üì (stored procedure)
Database (MySQL / SQL Server)
```

## Common Patterns

### ViewModel Constructor Pattern
```csharp
public ViewModel_MyFeature(
    IService_MyService service,
    IService_ErrorHandler errorHandler,
    IService_LoggingUtility logger) : base(errorHandler, logger)
{
    _service = service;
}
```

### Service Implementation Pattern
```csharp
public class Service_MyService : IService_MyService
{
    private readonly Dao_MyEntity _dao;
    
    public Service_MyService(Dao_MyEntity dao)
    {
        _dao = dao;
    }
    
    public async Task SaveAsync(Model_MyEntity entity)
    {
        return await _dao.SaveAsync(entity);
    }
}
```

using CommunityToolkit.Mvvm.ComponentModel;
using CommunityToolkit.Mvvm.Input;

namespace MyApp.ViewModels;

/// <summary>
/// ViewModel for [FEATURE_NAME]
/// </summary>
public partial class ViewModel_[FEATURE_NAME] : ViewModel_Shared_Base
{
    private readonly IService_[SERVICE_NAME] _service;

    [ObservableProperty]
    private string _statusMessage = "Ready";

    [ObservableProperty]
    private ObservableCollection<Model_[ENTITY]> _items;

    public ViewModel_[FEATURE_NAME](
        IService_[SERVICE_NAME] service,
        IService_ErrorHandler errorHandler,
        IService_LoggingUtility logger) : base(errorHandler, logger)
    {
        _service = service;
        Items = new ObservableCollection<Model_[ENTITY]>();
    }

    [RelayCommand]
    private async Task LoadDataAsync()
    {
        if (IsBusy) return;
        try
        {
            IsBusy = true;
            StatusMessage = "Loading...";

            var result = await _service.GetDataAsync();
            if (result.IsSuccess)
            {
                Items.Clear();
                foreach (var item in result.Data)
                {
                    Items.Add(item);
                }
                StatusMessage = $"Loaded {Items.Count} items";
            }
            else
            {
                _errorHandler.ShowUserError(
                    result.ErrorMessage,
                    "Load Error",
                    nameof(LoadDataAsync));
            }
        }
        catch (Exception ex)
        {
            _errorHandler.HandleException(
                ex,
                Enum_ErrorSeverity.Medium,
                nameof(LoadDataAsync),
                nameof(ViewModel_[FEATURE_NAME]));
        }
        finally
        {
            IsBusy = false;
        }
    }
}

Troubleshooting

‚ùå Copilot doesn't recognize my skill

Check these things:

  • File is named exactly SKILL.md (all caps)
  • Frontmatter has three dashes before and after (---)
  • The name field matches your folder name
  • Description includes keywords like "ViewModel", "MVVM", "architecture"
  • Restart VS Code to reload skills
‚ùå Skill activates but doesn't use scripts/references

Make sure you:

  • Reference files using relative paths in SKILL.md
  • Use clear link syntax: [text](./folder/file.ext)
  • Ask Copilot explicitly to "check the reference docs" or "run the script"
  • Keep file paths relative to the skill root (no absolute paths)
‚ùå SKILL.md is too long (over 500 lines)

Solution: Move content to references/

  • Keep high-level rules and examples in SKILL.md
  • Move detailed explanations to references/detailed-guide.md
  • Move violation examples to references/common-mistakes.md
  • Reference these files using relative links in SKILL.md
  • Copilot loads references on-demand, not automatically
‚ùå Scripts aren't executing

Common causes:

  • PowerShell execution policy blocks scripts
  • Script doesn't have executable permissions (Linux/Mac)
  • You need to explicitly ask Copilot to "run the validation script"
  • Script has syntax errors (test it manually first)

Fix: Run Set-ExecutionPolicy RemoteSigned -Scope CurrentUser (Windows)
