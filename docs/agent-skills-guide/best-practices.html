Best Practices

Write skills that activate reliably, stay maintainable, and scale with your repo

1) Write a Discovery-Ready Description (Most Important)

Copilot decides whether to load a skill primarily from only the name and description in your SKILL.md frontmatter.

Rule of thumb: If your skill doesn't activate reliably, improve the description first.

Use this formula

description: <WHAT it does>. Use when asked to <WHEN>. Supports <keywords/users say>.

Good example

description: Enforces MVVM architecture rules for WinUI 3 applications. Use when creating or modifying ViewModels, Services, DAOs, or XAML bindings. Prevents ViewModels from calling DAOs directly, requires x:Bind, and validates service-layer flow.

Poor example

description: MVVM helper

The poor example fails because it does not include real user triggers ("create viewmodel", "audit xaml", "fix mvvm violations"), and it doesn't describe capabilities.

2) Use Progressive Loading (Keep SKILL.md Short)

Keep SKILL.md focused. Move long content into references/ and link it from SKILL.md.

  • Put the "quick rules" and the most common workflow in SKILL.md
  • Put edge cases and deep patterns in references
  • Keep templates minimal and placeholder-driven
Practical target: Keep SKILL.md under ~500 lines. Add links to reference docs for the rest.

3) Bundle Resources the Right Way

Use the right folder for the right kind of content:

Folder Purpose Loads when‚Ä¶ Examples
scripts/ Deterministic automation Executed Validation scans, report generators, safe bulk edits
references/ Deep docs and rules Referenced Architecture rules, cheat sheets, troubleshooting
templates/ Editable scaffolds Referenced ViewModel template, DAO template, test template
assets/ Static resources used as-is Copied/embedded Logo, baseline image, report HTML shell
Tip: If Copilot should edit it, it belongs in templates/ not assets/.

4) Write Workflows as Numbered Steps

Skills work best when instructions are action-oriented and structured.

Example workflow format 
## Step-by-Step Workflow: Create a new ViewModel

1. Confirm the ViewModel inherits from ViewModel_Shared_Base and is partial.
2. Inject services (never DAOs) via constructor.
3. Add [ObservableProperty] fields for bindable state.
4. Add [RelayCommand] methods for actions.
5. Ensure async methods end with Async.
6. Confirm XAML uses x:Bind with explicit Mode.
7. If requested, scaffold from templates/viewmodel-template.cs.
8. If auditing, run scripts/validate-mvvm.ps1 and summarize output.

5) Use Consistent Naming

Consistency improves skill discovery and reduces confusion.

  • Use short, clear skill names: mvvm-architecture-enforcer, xaml-xbind-auditor
  • Use verbs in scripts: validate-, scan-, generate-
  • Use descriptive reference names: architecture-rules.md, common-violations.md
  • Add placeholders in templates: [FEATURE_NAME], [SERVICE_NAME]

6) Troubleshooting

‚ùå My skill never activates
  • Improve the description with concrete triggers and keywords
  • Ensure name matches folder name
  • Restart VS Code to refresh skill discovery
‚ùå My scripts aren't running
  • Make sure the user request includes running the script
  • Validate the script manually first
  • Check PowerShell execution policy on Windows
‚ùå Copilot ignores my templates
  • Link templates from SKILL.md using relative paths
  • Keep templates simple; avoid huge multi-file scaffolds unless necessary
  • Use placeholders and explain which parts Copilot should fill in
‚ùå My skill is too large / hard to maintain
  • Move long content to references/
  • Split workflows into separate reference pages
  • Keep SKILL.md as a "table of contents" + short checklist

7) Maintain Skills Like Code

Skills are part of your engineering system. Treat them like code:

  • Version and review changes in pull requests
  • Keep a short changelog in reference docs when behavior changes
  • Prefer small edits over rewrites
  • Update docs when you change architecture rules
